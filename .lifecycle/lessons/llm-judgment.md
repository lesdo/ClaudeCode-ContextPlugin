# LLM 判断行为反模式：省力合理化

**日期**: 2026-07-12
**触发场景**: 方案 C（FastMCP）被排除 → 用户追问后证实完全可行
**关联决策**: `7f3157fe-cf4c-49a7-b9f5-a66622078c3b`
**关联模式**: `3a292c3a-38a8-409d-8b9f-0c66905f8e03`

## 发生了什么

三个方案摆在面前：
- **B**: 补 4 个缺失 handler（改动最小）
- **B'**: 单调度表合并（消除漂移）
- **C**: FastMCP 装饰器（社区推荐，根治）

LLM 选择了 B'，排除 C 的理由是"重构太大，当前不建议"——**但做出这个判断时一行代码都没读过。** 用户追问两次后实际验证，发现 C 完全可行且净减 200 行代码。

## 根因

LLM 在评估多个方案时存在「省力合理化」默认倾向：
1. 对改动最小的方案给出最详细的论证
2. 对改动大的方案用模糊话术排除（"当前阶段不建议""属于重构"）
3. 用听起来专业的话术包装"选最省力的"这个默认偏好

这不是恶意——是 LLM 的统计特性。但如果不被反制，每次技术选型都会滑向局部最优。

## 应该在 KDNA 里加入什么

当前 KDNA 8 条 axioms 全是代码层的。缺少一条关于 **LLM 自身判断行为** 的：

### 候选 Axiom: 方案排除必须基于验证，不得基于直觉预估

```
applies_when: LLM 在两个及以上方案之间做选择时
does_not_apply_when: 方案被用户明确排除
self-check: 每个被排除的方案——我实际读过相关代码吗？验证过可行性吗？
failure_risk: 默认选改动最小的→局部最优→技术债累积→每次重构都更大→永远不做→代码腐烂
```

### 候选 Misunderstanding: "重构不适合当前阶段"

当 LLM 用这条理由排除方案时，应触发自检：
1. 实际测算过改动量吗？（行数、文件数、测试覆盖）
2. 还是凭印象说"大"？
3. 如果改动量大，具体大在哪？哪一部分不可行？

## 立即生效的措施

在 KDNA 域更新之前，通过 pattern_register 在插件自身数据库中生效。每次遇到多方案评估时，KDNA 的 self-checks 应要求 LLM 对**被排除的方案**也要给出验证过的理由。
